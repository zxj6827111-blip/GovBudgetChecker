## 一、修订结论摘要

本次复核后，原报告中“缓存与构建产物可清理”的方向是正确的，但存在以下关键修正：

1. `tests/`、`e2e/`、`samples/`、`pytest.ini` 不应按“可直接删除”处理；它们与当前测试链路或验收要求相关。

5. `engine/rules_v33_new.py` 作为死代码候选结论基本成立，可进入删除候选清单。

## 二、审核口径（避免“删对了文件却做错了事”）

| 口径 | 含义 | 处理策略 |
|------|------|----------|
| 仓库口径（Git） | 是否应保留在代码仓库，保障开发/测试/追溯 | 不轻易删除测试、样本、配置 |
| 部署口径（镜像/制品） | 是否应进入生产镜像或发布包 | 通过 `.dockerignore` / CI 打包规则排除 |
| 运行口径（运行数据） | 运行期生成物是否需要定期清理 | 建立保留天数与回收任务 |

## 四、当前体积热点（实测验证结果）

### 4.1 实测体积数据

| 目录 | 体积（实测） | 说明 |
|------|------------|------|
| `app/` | 467.72 MB | 主要由 `node_modules` 与 `.next` 构成 |
| `uploads/` | 151.79 MB | 运行期上传与任务产物，138 个任务目录 |
| `.mypy_cache/` | 64.45 MB | 类型检查缓存 |
| `samples/` | 5.05 MB | 测试/演示样本，建议保留在仓库 |
| `engine/` | 0.44 MB | 核心引擎代码 |
| `src/` | 0.35 MB | 源代码 |
| `services/` | 0.25 MB | 服务层代码 |

### 4.2 实测方法（PowerShell）

```powershell
# 获取各目录实际体积
Get-ChildItem -Path . -Directory | ForEach-Object {
    $size = (Get-ChildItem $_.FullName -Recurse -File -ErrorAction SilentlyContinue | Measure-Object -Property Length -Sum).Sum / 1MB
    [PSCustomObject]@{ Directory = $_.Name; SizeMB = [math]::Round($size, 2) }
} | Sort-Object SizeMB -Descending | Format-Table -AutoSize
```

### 4.3 samples/ 目录保留理由（实测验证）

经检查，`samples/` 目录被 `tests/test_samples.py` 测试用例直接依赖：

- `MANIFEST_PATH = Path("samples/manifest.yaml")` - 加载样本清单
- `test_manifest_meta_defaults()` - 验证清单与默认配置一致性
- `test_template_sample_tables()` - 验证模板样本的表格结构
- `test_good_sample_expectations()` - 验证良好样本的期望结果
- `test_bad_sample_expectations()` - 验证问题样本的期望触发规则

**结论**：删除 `samples/` 会导致 `test_samples.py` 测试失败，必须保留在仓库中。

### 5.4 误删回滚方案

#### Git 恢复已提交的删除（推荐 `git restore`）

```bash
# 恢复单个文件
git restore --source=HEAD~1 -- <file_path>

# 恢复整个目录
git restore --source=HEAD~1 -- <directory_path>

# 查看最近提交的变更
git log --oneline -5
```

#### Git 恢复未提交的删除（仅在未提交时有效）

```bash
# 恢复单个文件
git restore -- <file_path>

# 查看当前被删除的跟踪文件
git ls-files -d

# 按路径逐个恢复（避免误覆盖其他未提交修改）
git restore -- <file_path>
```

> 注意：不建议使用 `git checkout -- .`，该命令会覆盖工作区其他未提交修改。

#### 手动备份恢复

删除前建议创建完整备份：

```powershell
# 创建备份目录
Copy-Item -Path . -Destination "GovBudgetChecker_backup_$(Get-Date -Format 'yyyyMMdd_HHmmss')" -Recurse
```

## 六、原报告漏检的关键点（新增）

### 6.1 运行数据保留策略（`uploads/`）

`uploads/` 不应简单删除，也不应无限增长。建议：

1. 保留最近 `N` 天（如 7/15/30 天）任务目录。  
2. 对已完成任务，只保留必要产物（最终报告、标注 PDF、导出 JSON/CSV），清理中间快照。  
3. 增加定时清理脚本（cron/CI 定时任务）。  
4. 在 API 层增加“删除任务及附件”管理接口（按任务状态限制）。  

### 6.2 敏感信息治理（高优先级）

当前风险：`.env.local`、`app/.env.local` 已被跟踪。  
仅在 `.gitignore` 增加规则不能解决历史跟踪问题，需：

1. 立即轮换已暴露的 API Key。  
2. 取消 Git 跟踪（保留本地文件）：
```bash
git rm --cached .env.local app/.env.local
git commit -m "security: untrack local env files"
```
3. 使用 `.env.example` 维护非敏感模板。  
4. 在 CI 增加密钥扫描（如 gitleaks/trufflehog）。  

### 6.3 忽略规则一致性

当前 `.gitignore` 中为 `pytest_cache/`，建议改为 `.pytest_cache/` 以避免遗漏。  
并补充：

- 根目录 `.dockerignore`
- `app/.dockerignore`（因为前端镜像 build context 为 `./app`）

## 十、最终修订意见

本项目当前不适合“按文件类型一刀切删除”。  
正确做法是：

1. **仓库保留能力资产**（测试、样本、质量配置）；  
2. **部署排除冗余载荷**（缓存、IDE、临时目录）；  
3. **运行期定期回收数据**（尤其 `uploads/`）；  
4. **优先处理安全问题**（已跟踪的本地环境变量文件）。
