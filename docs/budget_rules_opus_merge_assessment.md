# Opus 方案评审与融合结果

## 结论
- `预算检查规则设计方案.md` 的主框架是合理的，可直接作为预算规则体系的主骨架。
- 已将高价值内容融合进 `rules/budget_v3_3_draft.yaml`，并对少数高误报风险点做了降级或白名单处理。

## 可直接借鉴（已采纳）
- 元/万元混用识别与统一换算（`预算检查规则设计方案.md:244`）。
- 部门/单位双分支识别与规则分流（`预算检查规则设计方案.md:257`）。
- 九张表结构与空表说明约束（`预算检查规则设计方案.md:63`, `预算检查规则设计方案.md:185`）。
- 核心勾稽链路：
  - T1 收支总计平衡（`预算检查规则设计方案.md:56`）
  - T3 合计=基本+项目（`预算检查规则设计方案.md:93`）
  - T8 合计=人员+公用（`预算检查规则设计方案.md:70`）
  - T9 三公合计分解（`预算检查规则设计方案.md:83`）

## 需改造后借鉴（已调整）
- `±0.01元` 作为统一容差过严（`预算检查规则设计方案.md:61`）。
  - 已改为分层容差：同表强校验、跨单位换算校验、OCR回退校验。
- 机关运行经费必须等于表8公用经费（`预算检查规则设计方案.md:106`）。
  - 已保留为高价值对账，但建议作为“中高优先级软约束”，避免跨地区口径差导致误报。
- 占位符“一律 critical”（`预算检查规则设计方案.md:192`）。
  - 已拆分 hard-fail 与普通占位符，降低误报。
- 年份一致性（`预算检查规则设计方案.md:201`）已补白名单：
  - “截至日期”
  - 书名号中的历史年份引用
  - 立项依据/任务书语境
- 绩效资金 <= 项目支出（`预算检查规则设计方案.md:132`）保留，但作为中优先级规则，不做一票否决。

## 不建议原样采用
- 把大量 NLP 表文勾稽直接放在第一阶段硬拦截（`预算检查规则设计方案.md:317`）。
  - 建议先上线稳定的结构+数值勾稽，再逐步放开 NLP 严格度。
